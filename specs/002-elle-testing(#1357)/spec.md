## 功能模块 *(必填)*

### Module 1: 原子性（Atomicity）测试模块 (优先级: P1)

**目的**: 验证NeuG数据库事务的原子性保证，确保事务要么完全执行成功（全部提交），要么完全失败（全部回滚），不存在部分提交的中间状态。该模块通过设计各种事务中断场景，验证数据库在异常情况下能够正确回滚所有未提交的更改。

**优先级原因**: 原子性是ACID的基础属性，是事务正确性的前提。如果事务不能保证原子性，数据将处于不一致状态，后续的隔离性和一致性测试将失去意义。因此必须首先验证原子性。

**独立测试**: 可以通过执行包含多个操作的事务，在事务执行过程中模拟各种中断场景（如异常抛出、显式回滚、系统崩溃等），验证数据库能够正确回滚所有未提交的更改，确保数据状态与事务开始前一致。

**核心组件**:

1. **Atomicity-C测试器（Atomicity-Commit Tester）**: 检查每个成功提交的事务，确保客户端收到的每个成功提交消息对应的所有插入或修改的数据项在提交后都可见。该测试器验证已提交事务的所有修改（包括节点创建、属性更新、边创建等）在提交后能够被后续查询正确读取。

2. **Atomicity-RB测试器（Atomicity-Rollback Tester）**: 检查每个中止的事务，确保其所有修改都不可见。该测试器验证已回滚或中止的事务的所有修改（包括节点创建、属性更新、边创建等）在事务中止后不会出现在数据库中，确保数据库状态与事务开始前一致。

**执行流程**:

测试按以下步骤执行：

1. **加载测试图**: 加载一个包含`Person`节点的图，每个节点具有唯一的`id`属性和一组`emails`属性

2. **初始化计数器**: 客户端执行一次全图扫描，统计图中的节点数量、边数量和emails数量，使用统计结果初始化计数器`committed`

3. **执行事务测试**: 执行N个所需测试的事务实例，对于每个成功提交的事务，将`committed`计数器递增

4. **最终状态扫描**: 重复执行全图扫描，统计最终的节点数量、边数量和emails数量，将结果存储在变量`finalState`中

5. **异常检查**: 执行异常检查，验证`committed=finalState`是否成立。如果相等，说明所有成功提交的事务的修改都正确可见（Atomicity-C测试），或所有中止的事务的修改都正确不可见（Atomicity-RB测试）

### Module 2: 隔离性（Isolation）测试模块 (优先级: P2)

**目的**: 验证NeuG数据库事务的隔离性保证，检测各种隔离异常（Isolation Anomalies），包括脏写（Dirty Write）、脏读（Dirty Read）、切割异常（Cut Anomalies）、观察事务消失（Observed Transaction Vanishes）、断裂读（Fractured Read）、丢失更新（Lost Update）、写偏斜（Write Skew）等。该模块使用Elle算法的环检测机制检测部分异常，同时使用自定义查询语言检测其他异常。

**优先级原因**: 隔离性是并发控制的核心，直接影响数据库的并发性能和正确性。NeuG数据库在Service模式下声称提供可串行化隔离级别，需要通过全面的隔离性测试验证其实现是否正确。该模块依赖Module 1的操作记录能力，但可以并行开发测试用例。

**独立测试**: 可以通过构造已知的隔离异常模式（如脏写、脏读、写偏斜等），在NeuG数据库上执行并发事务，使用Elle算法的环检测或自定义查询验证是否出现预期的隔离异常。

**核心组件**:

1. **Dirty Write测试器（Dirty Write Tester）**: 检测脏写异常，验证两个并发事务不能同时修改同一数据对象。该测试器通过Elle算法的环检测机制，在依赖图中检测包含WW（写写）边的环来识别G0异常（脏写）。

2. **Dirty Read测试器（Dirty Read Tester）**: 检测脏读异常，验证事务不能读取其他未提交事务写入的数据。该测试器通过分析操作历史，检测事务是否读取了未提交事务写入的数据值。

3. **Cut Anomalies测试器（Cut Anomalies Tester）**: 检测切割异常，验证事务读取的数据集是否被其他事务的提交操作"切割"。该测试器使用自定义查询语言，检测事务在多次读取同一数据集时，是否出现结果被其他事务的提交操作分割的情况。

4. **Observed Transaction Vanishes测试器（Observed Transaction Vanishes Tester）**: 检测观察事务消失异常，验证已观察到的数据是否会在后续读取中消失。该测试器使用自定义查询语言，检测事务在第一次读取中观察到的数据，是否在后续读取中消失（例如节点被删除）。

5. **Fractured Read测试器（Fractured Read Tester）**: 检测断裂读异常，验证事务读取的数据是否来自多个不同版本的不一致快照。该测试器使用自定义查询语言，检测事务在读取多个数据项时，这些数据项是否来自不同的事务版本，导致读取结果不一致。

6. **Lost Update测试器（Lost Update Tester）**: 检测丢失更新异常，验证并发事务的更新操作是否被正确序列化。该测试器通过分析操作历史，检测多个事务对同一数据对象的更新操作是否出现丢失，导致最终结果不正确。

7. **Write Skew测试器（Write Skew Tester）**: 检测写偏斜异常，验证事务基于读取值进行写入时是否出现不一致。该测试器通过Elle算法的环检测机制，在依赖图中检测包含RW（读写）边的反依赖环来识别G2异常（写偏斜）。

**执行流程**:

测试按以下步骤执行：

1. **加载测试图**: 将测试图加载到数据库中，测试图包含节点、边和属性等数据项

2. **配置客户端**: 配置指定数量的写客户端和读客户端，用于执行并发事务测试

3. **执行事务序列**: 写客户端和读客户端在配置的时间段内执行一系列事务。每个事务由以下部分组成：
   - **BEGIN操作**: 事务开始
   - **有序的读写操作序列**: 对任意数据项集合（节点、边、属性）执行读写操作
   - **COMMIT或ABORT操作**: 事务提交或中止
   - 每个写操作创建一个数据项版本，版本被分配一个来自全序集合的唯一时间戳（如自然数），版本i的数据项x表示为xi
   - 所有数据项都有一个由初始事务T产生的初始未出生版本，位于每个数据项版本顺序的开始

4. **记录操作历史**: 记录执行历史H，包括：
   - 每个事务的读写操作
   - 读取和写入的数据项版本
   - 提交或中止操作

5. **构建依赖关系**: 从历史H中识别三种类型的直接依赖关系：
   - **读依赖（Read-Depends, wr）**: 事务Tj直接读依赖于Ti，如果Ti写入某个版本xk且Tj读取xk
   - **反依赖（Anti-Depends, rw）**: 事务Tj直接反依赖于Ti，如果Ti读取版本xk且Tj写入xk在版本顺序中的下一个版本
   - **写依赖（Write-Depends, ww）**: 事务Tj直接写依赖于Ti，如果Ti写入版本xk且Tj写入xk在版本顺序中的下一个版本

6. **构建直接序列化图**: 从历史H构建直接序列化图DSG(H)：
   - 图中的每个节点对应一个已提交的事务
   - 图中的边对应事务之间的直接冲突类型（读依赖、反依赖、写依赖）

7. **收集读客户端结果**: 执行期结束后，收集所有读客户端的执行结果

8. **执行全图扫描**（某些测试需要）: 对于需要额外信息的测试，在执行期结束后执行一次全图扫描，收集异常检查所需的信息

9. **异常检查**: 基于DSG(H)的属性进行异常检测：
   - 使用Elle算法的环检测机制检测依赖图中的环，识别可通过环检测发现的异常（如G0、G1、G2系列异常）
   - 使用自定义查询语言检测无法通过环检测发现的异常（如Fractured Read、Observed Transaction Vanishes等）

10. **扩展谓词操作**（如适用）: 对于基于谓词的操作，当事务基于条件P执行读写时，数据库为P适用的每个数据项选择一个版本，称为谓词的版本集Vset(P)。如果事务Ti覆盖了Vset(Pi)中的版本，则事务Tj改变基于谓词的读操作ri(Pi)的匹配结果

### Module 3: 一致性和持久性（Consistency and Durability）测试模块 (优先级: P3)

**目的**: 验证NeuG数据库事务的一致性和持久性保证。一致性测试验证数据库在事务执行前后始终满足所有定义的约束条件（如主键约束、外键约束、唯一性约束、数据类型约束等）。持久性测试验证已提交事务的更改在系统崩溃、重启等故障场景下能够正确恢复，不会丢失数据。

**优先级原因**: 一致性和持久性是数据完整性和可靠性的基础。一致性确保数据始终符合业务规则，持久性确保数据不会因系统故障而丢失。该模块可以与前两个模块并行开发，但需要完整的数据库环境支持。

**独立测试**: 可以通过构造违反约束条件的事务操作，验证数据库能够拒绝这些操作并保持一致性。可以通过模拟系统崩溃场景，验证已提交事务的更改在恢复后能够正确持久化。

**核心组件**:

1. **Durability测试器（Durability Tester）**: 验证已提交事务的更改在系统崩溃、重启等故障场景下能够正确恢复，不会丢失数据。该测试器通过执行工作负载、模拟系统崩溃、重启数据库并验证数据恢复来测试持久性保证。

2. **Consistency测试器（Consistency Tester）**: 验证数据库是否满足线性一致性（Linearizability）保证。在线性一致性模型下，一旦某个更新操作（写操作）完成并通知了用户，那么任何用户在任何时间、任何节点上发起的读操作，都必须返回这个最新的值。该测试器通过构造并发读写操作序列，验证数据库在所有操作的时间线上是否存在一个全局一致的顺序，使得每个读操作都能看到在其之前完成的写操作的最新结果，确保系统对外呈现单一、一致的数据视图。

**执行流程**:

Durability测试按以下步骤执行：

1. **配置工作负载**: 使用LDBC工作负载驱动执行SNB Interactive工作负载。数据库和工作负载驱动必须配置为与性能运行相同的方式

2. **预热阶段**: 数据库经历一个预热期，确保系统达到稳定状态

3. **执行工作负载**: 执行2小时的模拟执行，期间数据库处理各种事务操作

4. **终止数据库进程**: 在2小时执行后，通过以下方式之一终止数据库进程：
   - 断开整个机器的连接（如果可能）
   - 硬进程终止（kill -9）
   - 注意：在云测试中，关闭机器可能不可行，需要使用其他终止方式

5. **重启数据库系统**: 重启数据库系统，使其从持久化存储中恢复

6. **验证数据恢复**: 每个客户端发出读取请求，查询它最后收到成功提交消息的实体（节点或边）的值。该读取操作应该产生正确的响应，验证已提交事务的更改已正确持久化

Consistency测试按以下步骤执行：

1. **约束支持检查**: 检查数据库系统支持的约束类型：
   - **关系数据库约束**: 主键约束、外键约束、列值的域约束、行内简单约束
   - **图数据库约束**: 通常支持有限，可能包括域约束和主键约束（如果支持索引）。图数据库系统可能在未来支持图特定约束，如：
     - 基于属性图的模式合规性（部分或完全）
     - 指导标签存在的规则
     - 结构图约束（如图的连通性、无环性）
     - 任意良构性约束

2. **约束验证测试**: 对于数据库支持的约束类型，执行以下测试：
   - 尝试违反约束的操作，验证数据库是否正确拒绝这些操作
   - 验证事务执行后数据库状态满足所有定义的约束条件
   - 如果违反约束，验证事务是否被正确拒绝并回滚

3. **约束范围说明**: 对于图数据库系统，如果约束支持有限或不在当前测试范围内，应在测试报告中明确说明约束支持的范围和限制
